# React Native 中高级开发工程面试题与答案

本文档用于准备 React Native 中级、高级开发工程师面试，重点覆盖 React Native 基础、React 原理、跨端机制、新架构、性能优化、原生集成、工程化、稳定性、测试发布和架构设计。

建议学习方式：

1. 先通读题目，判断自己能否在 2 分钟内说出清晰答案。
2. 第二遍只看题目，按“结论、原理、项目经验、坑点”组织表达。
3. 高级题要重点练取舍：为什么这么做，不这么做会有什么问题。
4. 场景题不要只背方案，要能结合业务规模、团队能力、端侧限制和上线风险说明落地路径。

适用范围：

- 中级：能独立完成 React Native 业务开发，理解 React Hooks、组件渲染、列表、网络、导航和基本调试。
- 高级：能定位性能、内存、卡顿和构建问题，能封装通用组件、接入原生能力并推动工程规范。
- 资深倾向：能做跨端架构设计、新架构迁移、稳定性治理、发布体系和团队协作方案。

## 一、中级 React Native 面试题

### 1. React Native 和原生开发的核心区别是什么

**答案：**

React Native 使用 JavaScript 或 TypeScript 描述界面和业务逻辑，通过 React 的声明式模型驱动 UI 更新，最终渲染为 iOS / Android 原生控件或原生绘制体系。

和纯原生开发相比：

- 开发语言和 UI 描述方式不同，RN 主要写 JS / TS 和 JSX。
- 跨端复用度更高，但平台差异仍然存在。
- 性能瓶颈经常出现在 JS 线程、UI 线程、桥通信、列表渲染和图片内存。
- 原生能力仍需要 Native Module、Native Component 或第三方库接入。

面试中要避免把 RN 说成“网页套壳”。RN 不是 WebView，它最终使用原生能力渲染和交互。

### 2. React Native 的主要线程有哪些

**答案：**

常见线程包括：

- JS Thread：执行 JavaScript 逻辑、React 渲染计算、状态更新、事件处理。
- UI / Main Thread：负责原生 UI 渲染、手势响应、布局提交等。
- Shadow Thread：旧架构中负责 Yoga 布局计算。
- Native Modules Thread：部分原生模块会在独立线程或线程池执行。

如果 JS Thread 被长任务阻塞，会出现点击无响应、动画卡顿、列表滑动掉帧等问题。如果 UI Thread 过载，会出现原生渲染层面的卡顿。

### 3. React Native 中 JS 线程阻塞会带来什么问题

**答案：**

JS 线程阻塞会导致：

- 事件响应延迟，比如按钮点击、输入框输入变慢。
- JS 驱动动画卡顿。
- 状态更新和页面渲染延迟。
- 定时器、Promise 回调、网络响应处理堆积。

常见原因包括：

- 大量同步计算。
- 一次性渲染过多组件。
- JSON 大对象解析。
- 日志过多。
- 在 render 阶段做复杂逻辑。

解决方式是拆分任务、缓存计算结果、使用虚拟列表、避免不必要渲染，把重计算放到原生、JSI、Worker 类方案或后端处理。

### 4. JSX 最终会变成什么

**答案：**

JSX 是语法糖，最终会被 Babel 转换为 `React.createElement` 或新的 JSX runtime 调用。

例如：

```tsx
<Text style={styles.title}>Hello</Text>
```

可以理解为创建一个 React Element：

```tsx
React.createElement(Text, { style: styles.title }, 'Hello')
```

React Element 只是描述 UI 的普通对象，不是真实原生控件。React 会根据组件树、状态变化和调和结果决定如何更新最终界面。

### 5. React Native 里的 `View`、`Text`、`Image` 是什么

**答案：**

它们是 React Native 提供的基础跨端组件：

- `View` 对应通用容器。
- `Text` 对应文本渲染。
- `Image` 对应图片渲染。

这些组件不是 DOM 元素，而是映射到平台原生视图或原生渲染能力。比如 iOS 上可能对应 UIKit 相关视图，Android 上对应 View 体系。

需要注意：RN 没有 CSS DOM 盒模型，布局主要由 Yoga Flexbox 实现，样式能力和 Web CSS 有相似但不完全一致的地方。

### 6. React Native 的样式和 Web CSS 有哪些差异

**答案：**

主要差异：

- RN 样式是 JS 对象，不是 CSS 文件里的选择器规则。
- 属性名使用驼峰，如 `backgroundColor`。
- 默认布局方向是 `flexDirection: 'column'`，Web Flex 默认是 row。
- 不支持完整 CSS 选择器、伪类、媒体查询。
- 长度默认是逻辑像素，不写 `px`。
- 样式没有继承体系，只有部分文本属性在 `Text` 内部有类似继承效果。

示例：

```tsx
const styles = StyleSheet.create({
  container: {
    flex: 1,
    padding: 16,
    backgroundColor: '#fff',
  },
});
```

### 7. `StyleSheet.create` 有什么作用

**答案：**

`StyleSheet.create` 的作用主要是：

- 统一组织样式，提升可读性。
- 在开发阶段做样式校验。
- 让样式对象引用稳定，减少内联对象导致的无意义变化。

早期 RN 中样式可能有注册优化，现在更重要的是工程规范和引用稳定性。面试中不要过度神化它，不能简单说“用了就一定性能更好”。

### 8. RN 中如何处理不同屏幕尺寸适配

**答案：**

常见手段：

- 使用 Flexbox 做弹性布局。
- 使用 `useWindowDimensions` 获取窗口尺寸，避免只在启动时读取。
- 使用百分比、`aspectRatio`、`minWidth`、`maxWidth` 控制布局。
- 使用 `PixelRatio` 处理像素密度相关问题。
- 通过设计 token 管理间距、字号、圆角。
- 对平板、折叠屏、横屏做断点布局。

不建议简单全局按屏幕宽度缩放所有字体和尺寸，因为可读性、系统字体缩放、无障碍和不同设备比例都会出问题。

### 9. `ScrollView` 和 `FlatList` 有什么区别

**答案：**

`ScrollView` 会一次性渲染所有子节点，适合内容少、结构固定的页面。

`FlatList` 基于 `VirtualizedList`，只渲染可视区域附近的元素，适合长列表、大数据列表。

选择原则：

- 少量静态内容：`ScrollView`。
- 长列表、分页列表、需要上拉加载：`FlatList`。
- 分组列表：`SectionList`。
- 极高性能列表：可考虑 FlashList 或 RecyclerListView。

### 10. `FlatList` 常见性能优化有哪些

**答案：**

常见优化：

- 提供稳定的 `keyExtractor`。
- `renderItem` 使用 `useCallback`。
- item 组件使用 `React.memo`。
- 避免在 item 中创建大量内联对象和函数。
- 固定高度列表使用 `getItemLayout`。
- 合理配置 `initialNumToRender`、`windowSize`、`maxToRenderPerBatch`。
- 图片使用合适尺寸和缓存策略。
- 分页加载，避免一次塞入过多数据。
- 避免把整个大对象作为 item props 传入。

追问点：优化不是参数越大越好。窗口过大会占内存，窗口过小会白屏，批量渲染过大会阻塞 JS。

### 11. 为什么列表 item 不建议使用数组 index 作为 key

**答案：**

因为 index 不能稳定标识数据项。当列表插入、删除、排序时，index 会变化，React 可能复用错误组件实例，导致状态错乱、渲染异常或动画错误。

正确做法是使用业务唯一 id：

```tsx
<FlatList
  data={users}
  keyExtractor={(item) => item.id}
  renderItem={renderUser}
/>
```

只有完全静态、不增删排序的列表，index 才相对安全。

### 12. `useState` 和 `useRef` 的区别是什么

**答案：**

`useState` 用于管理会触发重新渲染的状态。

`useRef` 用于保存可变引用，修改 `ref.current` 不会触发重新渲染。

典型用法：

- `useState`：页面数据、开关状态、输入值。
- `useRef`：定时器 id、上一次值、组件实例引用、防重复点击标记。

如果某个值变化后 UI 需要更新，用 `useState`。如果只是跨渲染保存变量，用 `useRef`。

### 13. `useEffect` 和 `useLayoutEffect` 有什么区别

**答案：**

`useEffect` 在渲染提交后异步执行，适合网络请求、订阅、日志等副作用。

`useLayoutEffect` 在布局和绘制前同步执行，适合需要立即读取布局并同步更新 UI 的场景。

在 RN 中，大多数业务副作用使用 `useEffect`。`useLayoutEffect` 如果执行重逻辑，会阻塞界面提交，应该谨慎使用。

### 14. `useMemo` 和 `useCallback` 是否一定能提升性能

**答案：**

不一定。

`useMemo` 缓存计算结果，`useCallback` 缓存函数引用。它们本身也有依赖比较和内存成本。

适合使用的场景：

- 计算确实比较重。
- 子组件使用 `React.memo`，需要稳定 props。
- Hook 依赖数组需要稳定引用。
- 列表 item 的 `renderItem`、`keyExtractor` 等希望稳定。

不适合所有地方机械添加，否则会增加复杂度，甚至让性能更差。

### 15. React 组件重新渲染的常见原因有哪些

**答案：**

常见原因：

- 本组件 state 改变。
- 父组件重新渲染，导致子组件也重新渲染。
- Context value 改变。
- props 引用变化。
- 外部 store 订阅数据变化。

优化思路：

- 缩小 state 影响范围。
- 拆分组件。
- 使用 `React.memo`。
- 稳定函数和对象引用。
- Context 拆分，避免一个大 Context 管全部状态。
- 对列表和复杂组件做按需订阅。

### 16. `React.memo` 的作用和局限是什么

**答案：**

`React.memo` 用于缓存函数组件渲染结果，当 props 浅比较没有变化时跳过重新渲染。

局限：

- 默认只做浅比较。
- props 中如果每次传入新对象、新数组、新函数，仍会重新渲染。
- 组件内部 state 或 Context 变化仍会触发渲染。
- 比较本身有成本，不适合滥用。

适合列表 item、纯展示组件、渲染成本高且 props 稳定的组件。

### 17. RN 中如何做页面导航

**答案：**

常见方案是 React Navigation，也可能使用 React Native Navigation 等更偏原生的方案。

React Navigation 常见导航器：

- Stack Navigator：页面压栈跳转。
- Tab Navigator：底部或顶部标签。
- Drawer Navigator：抽屉导航。

工程上要关注：

- 路由类型定义。
- 页面参数校验。
- 深链 Deep Link。
- 登录态路由守卫。
- 页面生命周期和埋点。
- Native Stack 与 JS Stack 的性能差异。

### 18. RN 中如何处理安全区域

**答案：**

一般使用 `react-native-safe-area-context`：

```tsx
import { SafeAreaProvider, useSafeAreaInsets } from 'react-native-safe-area-context';
```

通过 `SafeAreaProvider` 提供上下文，用 `SafeAreaView` 或 `useSafeAreaInsets` 处理刘海屏、底部 Home Indicator、状态栏等区域。

不要简单写死顶部高度，因为不同设备、横竖屏、沉浸式状态栏和 Android 厂商行为都不同。

### 19. RN 中如何处理键盘遮挡

**答案：**

常见方式：

- `KeyboardAvoidingView`。
- `ScrollView` 配合 `keyboardShouldPersistTaps`。
- 监听 `Keyboard` 事件手动调整布局。
- 使用成熟表单和键盘处理库。
- iOS 和 Android 分平台处理 `behavior`。

需要关注输入框聚焦、底部按钮、聊天输入栏、弹窗表单和横屏场景。中高级候选人应能说明自己遇到过的平台差异。

### 20. RN 中网络请求通常怎么封装

**答案：**

常见封装层次：

- 底层请求库，如 `fetch` 或 `axios`。
- 统一 baseURL、超时、headers。
- 请求和响应拦截器。
- token 注入和刷新。
- 错误码统一处理。
- 重试、取消请求、幂等保护。
- 数据缓存，如 React Query、SWR、自研 store。
- 业务 API 方法层。

高级回答要提到：请求封装不要直接绑死 UI，错误处理要区分网络错误、业务错误、登录失效、服务端异常和用户取消。

## 二、React Native 原理与机制题

### 21. React Native 旧架构的 Bridge 是什么

**答案：**

旧架构中，JS 和 Native 运行在不同上下文里，通过 Bridge 通信。JS 侧调用原生模块时，会把调用信息序列化后传给 Native；Native 回调 JS 时也需要经过类似流程。

Bridge 的特点：

- 通信通常是异步的。
- 数据需要序列化和反序列化。
- 高频、小粒度通信成本高。
- 大对象传输会增加性能压力。

所以在旧架构下，应该减少 JS 和 Native 的频繁往返，尽量批量通信，把高频动画、手势、传感器处理放到更靠近原生或 UI 线程的位置。

### 22. JSI 是什么，解决了什么问题

**答案：**

JSI 是 JavaScript Interface，是 RN 新架构中的底层接口，让 C++ 对象和 JS Runtime 可以更直接地交互。

它解决的问题：

- 减少 Bridge 序列化成本。
- 支持同步调用某些原生能力。
- 允许 JS 持有 HostObject / HostFunction。
- 为 TurboModules、Fabric、Reanimated 等高性能能力提供基础。

JSI 不是业务层 API，而是底层能力。面试中要说清楚它降低通信成本，但不代表所有调用都应该同步化。

### 23. TurboModules 和传统 Native Modules 有什么区别

**答案：**

TurboModules 是新架构下的原生模块系统。

主要区别：

- 基于 JSI，通信路径更短。
- 支持懒加载，不一定启动时全部初始化。
- 通过 Codegen 生成类型安全的绑定代码。
- 更适合跨平台一致接口和大规模模块治理。

传统 Native Modules 更多依赖 Bridge，初始化和通信成本更高，类型约束也弱一些。

### 24. Fabric 是什么

**答案：**

Fabric 是 RN 新架构中的渲染系统，用于替代旧架构的 UI Manager 渲染链路。

核心特点：

- 使用 C++ Shadow Tree。
- 更好地支持同步布局和并发渲染。
- 与 React 18 并发能力更契合。
- 和 JSI、Codegen、TurboModules 协同工作。
- 提升原生组件创建、更新和布局链路的一致性。

Fabric 不等于“自动让所有页面变快”。迁移后仍需要治理组件渲染、列表、图片和业务逻辑。

### 25. Codegen 在 RN 新架构中有什么作用

**答案：**

Codegen 根据 JS / TS 侧声明生成原生绑定代码，让 JS 和 Native 的接口更类型安全、更一致。

作用包括：

- 生成 TurboModule 接口绑定。
- 生成 Fabric Native Component 绑定。
- 减少手写桥接代码。
- 在编译期暴露接口不匹配问题。

工程上要关注 schema 定义、生成脚本、iOS / Android 编译链路，以及版本升级带来的生成代码变化。

### 26. Hermes 是什么，为什么很多 RN 项目会启用它

**答案：**

Hermes 是 Meta 为 React Native 优化的 JavaScript 引擎。

常见优势：

- 启动速度更好。
- 内存占用通常更低。
- 支持预编译字节码。
- 和 RN 生态集成更紧密。
- 便于采集 JS 性能 profile。

需要注意：启用 Hermes 后，某些依赖如果使用了不兼容的 JS 特性或原生调试方式，可能需要适配。高级候选人应能说明 Hermes、JSC、V8 的选型取舍。

### 27. Yoga 在 RN 中负责什么

**答案：**

Yoga 是 RN 使用的跨平台布局引擎，负责按照 Flexbox 规则计算布局。

它根据组件的样式、父子关系、尺寸约束计算每个节点的位置和大小，然后把布局结果提交给原生渲染层。

常见相关问题：

- `flexDirection` 默认是 column。
- `alignItems` 和 `justifyContent` 方向容易混。
- 文本、图片、绝对定位会影响布局结果。
- 过深组件树和复杂布局会增加计算成本。

### 28. Metro 是什么

**答案：**

Metro 是 React Native 默认的 JavaScript bundler。

它负责：

- 解析模块依赖。
- 转换 TS / JSX / 新语法。
- 打包 JS bundle。
- 提供开发服务器和热更新能力。
- 处理资源文件引用。

工程上常见配置包括路径别名、monorepo 支持、资源扩展名、Babel 插件、环境变量和 sourcemap 生成。

### 29. RN 的热重载、Fast Refresh 和 OTA 热更新有什么区别

**答案：**

Fast Refresh 是开发期能力，用于保存代码后快速刷新界面，并尽量保留组件状态。

OTA 热更新是生产发布能力，把 JS bundle 和资源下发到用户设备，绕过应用商店重新发版流程的一部分。

区别：

- Fast Refresh 用于本地开发。
- OTA 用于线上修复或小功能发布。
- OTA 不能随意修改原生代码、权限、原生依赖和二进制能力。
- OTA 要遵守平台规则、灰度策略和回滚机制。

### 30. RN 中事件从原生传到 JS 大概经历什么流程

**答案：**

以点击事件为例：

1. 原生视图接收到触摸事件。
2. RN 事件系统将事件包装为 JS 可识别的数据。
3. 事件被分发到对应组件的 JS 回调。
4. JS 执行业务逻辑，可能触发 state 更新。
5. React 重新计算组件树差异。
6. 更新指令提交到原生 UI。

旧架构下事件跨 Bridge，有序列化成本。新架构下事件链路更直接，但 JS 线程阻塞仍会影响响应。

## 三、性能优化与稳定性题

### 31. RN 页面卡顿你会怎么排查

**答案：**

排查思路：

1. 判断卡顿发生在启动、进入页面、滚动、点击、动画还是网络返回后。
2. 区分 JS Thread 卡顿还是 UI Thread 卡顿。
3. 使用 React DevTools Profiler、Hermes Profile、Xcode Instruments、Android Studio Profiler、Systrace / Perfetto 等工具。
4. 检查组件重复渲染、列表参数、图片尺寸、同步计算、日志和大 JSON。
5. 检查原生侧主线程任务、布局层级、阴影圆角、图片解码和桥通信。
6. 做小步验证，每次只改一个变量。

高级回答要强调：先定位瓶颈，再优化。不要凭感觉到处加 memo。

### 32. RN 首屏加载慢怎么优化

**答案：**

常见优化方向：

- 减少启动阶段 JS bundle 体积。
- 延迟加载非首屏模块。
- 控制启动时全局初始化数量。
- 原生启动页和 RN 首屏衔接优化。
- 减少首屏接口数量，做接口聚合或缓存。
- 首屏骨架屏和渐进渲染。
- 启用 Hermes 并关注字节码产物。
- 图片预加载和本地兜底资源。
- 避免启动时同步读取大量本地存储。

需要结合数据衡量，比如冷启动耗时、JS bundle 执行耗时、首屏可交互时间。

### 33. RN 内存占用高怎么排查

**答案：**

排查方向：

- 图片是否过大、是否有缓存失控。
- 列表窗口是否过大，item 是否保留过多复杂视图。
- 全局 store 是否存了大量无用数据。
- 事件监听、定时器、订阅是否未释放。
- Native Module 是否持有对象不释放。
- WebView、地图、视频等重组件是否复用或及时销毁。
- 是否存在闭包捕获大对象。

工具上 iOS 可用 Instruments Allocations / Leaks，Android 可用 Memory Profiler / LeakCanary，JS 可结合 Hermes heap snapshot 或业务埋点。

### 34. RN 图片加载有哪些性能问题

**答案：**

常见问题：

- 原图尺寸远大于显示尺寸，导致解码和内存成本高。
- 长列表中大量图片同时加载。
- 没有缓存策略，反复请求。
- base64 图片过大，增加 JS 内存和 bundle 体积。
- GIF 或大动图占用高。
- iOS / Android 缓存行为差异。

优化方式：

- 服务端按需裁剪图片尺寸。
- 使用缩略图和渐进加载。
- 列表中控制并发加载。
- 使用成熟图片库和缓存策略。
- 避免把大图片直接打进 JS bundle。

### 35. 动画卡顿如何处理

**答案：**

首先判断动画是 JS 驱动还是原生 / UI 线程驱动。

优化方向：

- 简单动画优先使用 `Animated` 的 native driver。
- 复杂手势和高频动画可使用 Reanimated。
- 避免动画过程中频繁 setState。
- 避免每帧跨 JS 和 Native 通信。
- 减少动画节点上的阴影、模糊、大图等重绘成本。
- 列表滚动联动动画要控制计算量。

高级候选人应能解释：如果动画每一帧都依赖 JS 线程计算，JS 一忙就会掉帧。

### 36. 如何避免重复渲染

**答案：**

常见做法：

- 拆分组件，让 state 尽量靠近使用位置。
- 使用 `React.memo` 包裹纯展示组件。
- 用 `useMemo` / `useCallback` 稳定复杂 props。
- 避免在 JSX 中创建新数组、新对象、新函数传给深层组件。
- Context 拆分，避免全局状态变化带动大面积重渲染。
- 使用按 selector 订阅的状态库。
- 列表 item 只传必要字段。

排查时可以使用 React DevTools Profiler 或 why-did-you-render 类工具辅助定位。

### 37. 大 JSON 数据处理导致卡顿怎么办

**答案：**

可选方案：

- 后端分页、裁剪字段、压缩数据。
- 避免一次性 JSON.parse 超大内容。
- 分批处理数据，让出事件循环。
- 把复杂计算放到服务端、原生或 JSI 模块。
- 使用缓存和增量更新。
- UI 先展示关键字段，非关键字段延迟处理。

核心思路是不要让 JS 线程长时间同步执行不可中断任务。

### 38. 如何设计 RN 崩溃和异常监控

**答案：**

需要同时覆盖 JS 和 Native：

- JS Error Boundary 捕获渲染错误。
- 全局 JS 异常处理。
- Promise unhandled rejection 监控。
- 原生 crash 采集，如 iOS crash、Android crash / ANR。
- Source map 上传，用于还原 JS 堆栈。
- 版本、bundle 版本、设备、系统、页面路径、用户操作上下文。
- 告警、聚合、去重和影响面评估。

高级回答要提到：线上排障必须能区分二进制版本和 JS bundle 版本，否则 OTA 后问题难定位。

### 39. 如何处理 RN 白屏问题

**答案：**

白屏常见原因：

- JS bundle 加载失败。
- JS 执行异常。
- 首屏接口阻塞且没有兜底 UI。
- Native 初始化 RN 容器失败。
- OTA 包损坏或版本不匹配。
- 路由初始化异常。
- 资源缺失。

治理方式：

- 原生容器设置超时兜底。
- JS 入口加 Error Boundary。
- 首屏骨架和错误页。
- OTA 校验、回滚和灰度。
- 启动关键路径日志。
- 监控首屏渲染成功率。

### 40. RN 中如何做性能埋点

**答案：**

常见指标：

- App 冷启动、热启动。
- RN 容器初始化耗时。
- JS bundle 加载和执行耗时。
- 页面首屏渲染耗时。
- 页面可交互耗时。
- 接口耗时。
- 列表滚动 FPS。
- JS 卡顿和 Native 卡顿。
- 内存峰值。

实现上可以在 Native 和 JS 两侧打点，统一 traceId 或页面 id，把关键阶段串起来。只在 JS 侧打点往往无法覆盖完整启动链路。

## 四、原生集成与跨端能力题

### 41. 如何封装一个 Native Module

**答案：**

传统方式大致步骤：

1. iOS 侧创建 Objective-C / Swift 模块并导出方法。
2. Android 侧创建 Kotlin / Java 模块并注册到 Package。
3. JS 侧通过 `NativeModules` 或封装层调用。
4. 对异步能力使用 Promise、Callback 或事件。
5. 处理线程切换、参数校验和错误码。

新架构下还要考虑 TurboModule、Codegen、类型声明和生成代码。

工程上建议 JS 层再包一层业务友好的 API，不要让业务到处直接调用原生模块。

### 42. Native Module 中 Promise、Callback、EventEmitter 怎么选

**答案：**

- Promise：适合一次性异步结果，如读取设备信息、请求权限、调用相机后返回结果。
- Callback：也能处理一次性或多次回调，但类型和生命周期更难管理。
- EventEmitter：适合持续事件流，如定位、蓝牙状态、网络变化、下载进度。

事件类能力要注意订阅和取消订阅，避免内存泄漏和重复回调。

### 43. 原生模块方法应该在哪个线程执行

**答案：**

取决于任务类型：

- UI 操作必须在主线程。
- 文件、网络、加解密、大计算不应阻塞主线程。
- 回调 JS 时要遵守 RN 模块线程规则。

面试中要说明：不能为了简单把所有原生模块都放主线程，否则容易造成掉帧和 ANR。也不能在后台线程直接操作 UI。

### 44. 如何封装一个 Native UI Component

**答案：**

适合需要原生视图能力的场景，比如地图、视频播放器、相机预览、复杂图表。

基本步骤：

- iOS / Android 分别实现原生 View。
- 暴露 props 给 JS。
- 暴露事件回调给 JS。
- JS 侧封装 React 组件。
- 处理生命周期、资源释放和平台差异。

新架构下还要考虑 Fabric Component、Codegen 和类型声明。

### 45. RN 与原生混合工程常见架构是什么

**答案：**

常见模式：

- 原生 App 作为壳，部分页面由 RN 实现。
- RN 页面由原生路由容器打开。
- JS bundle 可以内置，也可以 OTA 更新。
- 公共能力通过 Native Module 暴露，如登录态、埋点、网络、设备能力。
- 页面间通过统一路由协议通信。

高级点在于边界设计：RN 不应直接依赖大量零散原生细节，原生也不应理解太多 JS 业务结构，应通过稳定协议隔离。

### 46. 如何处理 iOS 和 Android 平台差异

**答案：**

常见方式：

- `Platform.OS` 做少量分支。
- 使用 `.ios.tsx`、`.android.tsx` 文件拆分。
- 在原生模块内部屏蔽差异，JS 暴露统一接口。
- 用设计规范统一平台差异。
- 对权限、状态栏、键盘、返回键、字体、阴影做专项适配。

原则是：小差异可以在 JS 分支解决，大差异应抽象成平台实现，避免业务代码到处散落 `Platform.OS`。

### 47. Android 返回键如何处理

**答案：**

使用 `BackHandler` 监听硬件返回键：

```tsx
useEffect(() => {
  const sub = BackHandler.addEventListener('hardwareBackPress', () => {
    // 返回 true 表示自己处理
    return true;
  });
  return () => sub.remove();
}, []);
```

实际项目中通常与导航库结合：

- 栈内页面优先 pop。
- 弹窗、选择器、浮层优先关闭。
- 首页二次返回退出。
- 特殊页面拦截返回并二次确认。

### 48. RN 中权限申请如何设计

**答案：**

权限设计要同时考虑平台差异和业务体验：

- JS 层提供统一权限服务。
- iOS / Android 分别处理权限声明和申请。
- 区分未申请、已授权、拒绝、永久拒绝、受限等状态。
- 拒绝后给出合理引导，而不是无限弹窗。
- 权限申请尽量发生在用户触发相关功能时。
- 关注 Android 版本差异和 iOS 隐私权限文案。

高级候选人应能说明相机、相册、定位、通知权限的差异。

### 49. 如何在 RN 中接入推送通知

**答案：**

大体流程：

- 原生侧集成 APNs / FCM 或厂商推送。
- 获取 device token 或 push token。
- 上传 token 到服务端并绑定用户。
- 处理前台、后台、点击通知、冷启动通知。
- JS 侧封装通知事件和跳转协议。
- 做权限申请、开关管理、埋点和到达率统计。

难点在于多厂商、多状态、多启动路径，以及通知点击后的路由恢复。

### 50. RN 接入第三方 SDK 要注意什么

**答案：**

要关注：

- iOS Pod、Android Gradle 依赖版本冲突。
- 最低系统版本和架构支持。
- 隐私合规和权限声明。
- SDK 初始化时机。
- 线程和生命周期。
- JS API 封装是否稳定。
- 失败和降级策略。
- 包体积影响。
- ProGuard / R8 混淆规则。

不要只看 Demo 能跑，还要看线上稳定性、升级维护和合规风险。

## 五、工程化、测试与发布题

### 51. RN 项目如何组织目录结构

**答案：**

常见结构：

```text
src/
  api/
  assets/
  components/
  constants/
  hooks/
  navigation/
  screens/
  services/
  store/
  theme/
  types/
  utils/
```

目录设计原则：

- 按职责分层，避免所有东西都塞 components。
- 页面级代码和通用组件分开。
- 跨端能力放 services 或 native adapter。
- API、状态、UI、业务逻辑边界清晰。
- 大项目可按业务模块拆分 feature。

### 52. RN 中 TypeScript 有什么价值

**答案：**

TypeScript 的价值：

- 类型约束 props、state、API 数据。
- 导航参数类型安全。
- 减少运行时字段错误。
- 提升重构安全性。
- 改善 IDE 提示和团队协作。
- 为 Native Module / Codegen 提供更清晰接口。

高级使用包括：路由参数泛型、接口响应类型、领域模型、组件变体类型、状态机类型和严格空值检查。

### 53. 如何设计通用组件库

**答案：**

设计原则：

- 先沉淀高频稳定组件，不要过早抽象。
- 组件 API 要贴近业务使用方式。
- 支持主题、尺寸、状态、禁用、加载、错误等常见场景。
- 保持受控和非受控边界清晰。
- 样式 token 化，避免魔法值。
- 有文档、示例和快照测试。
- 注意无障碍和点击区域。

高级点：通用组件不要把业务逻辑塞进去，业务组件可以基于通用组件再封装。

### 54. RN 项目状态管理如何选型

**答案：**

常见选择：

- 本地 UI 状态：`useState`、`useReducer`。
- 跨组件轻量状态：Context、Zustand、Jotai。
- 大型可预测状态：Redux Toolkit。
- 服务端缓存状态：React Query / SWR。
- 表单状态：React Hook Form、Formik 等。

关键不是“用哪个库最强”，而是区分状态类型：

- UI 状态。
- 业务领域状态。
- 服务端缓存状态。
- 持久化状态。
- 临时流程状态。

不同状态混在一个全局 store 里，后期会很难维护。

### 55. Redux 在 RN 中适合什么场景

**答案：**

Redux 适合：

- 多页面共享复杂业务状态。
- 需要可预测状态流。
- 团队规模较大，需要统一规范。
- 需要中间件处理日志、持久化、异步流。
- 复杂业务需要可调试、可回放的状态变化。

不适合：

- 简单页面局部状态。
- 服务端数据缓存全部手写进 Redux。
- 为了“显得架构完整”而把所有状态都全局化。

现代项目更推荐 Redux Toolkit，减少样板代码。

### 56. 如何做 RN 单元测试和组件测试

**答案：**

常见工具：

- Jest：单元测试。
- React Native Testing Library：组件行为测试。
- Mock Service Worker 或自定义 mock：接口模拟。

测试重点：

- 工具函数、hooks。
- 复杂组件交互。
- 表单校验。
- 状态流 reducer。
- API 错误分支。
- 关键业务页面的渲染和点击行为。

组件测试不应过度依赖内部实现，而应验证用户可感知行为。

### 57. 如何做 RN E2E 测试

**答案：**

常见工具是 Detox，也可以结合 Appium。

适合覆盖：

- 登录注册主流程。
- 支付、下单、发布等核心路径。
- 深链打开。
- 权限弹窗。
- 升级和回归风险高的流程。

难点包括环境稳定性、测试数据准备、异步等待、真机和模拟器差异、CI 时间成本。E2E 测试应覆盖关键链路，不适合追求 100% 页面覆盖。

### 58. RN CI/CD 通常包含哪些环节

**答案：**

常见环节：

- 依赖安装和缓存。
- TypeScript 类型检查。
- ESLint / Prettier。
- 单元测试。
- iOS Pod 安装。
- Android Gradle 构建。
- iOS archive。
- 产物签名。
- sourcemap 上传。
- OTA bundle 构建和发布。
- 包体积检查。
- 自动上传测试平台或应用商店。

高级回答要提到环境一致性、证书管理、构建缓存、版本号策略和失败告警。

### 59. OTA 热更新发布要如何治理

**答案：**

治理要点：

- JS bundle 版本和 native binary 版本绑定。
- 灰度发布，按用户、设备、版本、渠道控制。
- 包完整性校验。
- 启动失败自动回滚。
- sourcemap 与 bundle 一一对应。
- 监控白屏率、崩溃率、关键业务指标。
- 变更范围评估，涉及原生能力变化必须发版。
- 保留紧急停止和回滚入口。

不要把 OTA 当成无限制发版，它更适合 JS 层 bug 修复、小范围 UI 和业务逻辑调整。

### 60. RN 多环境配置怎么做

**答案：**

常见环境：

- dev。
- test / staging。
- pre-release。
- production。

配置内容：

- API 域名。
- 资源域名。
- app id / scheme。
- 第三方 SDK key。
- 日志开关。
- feature flag。
- OTA deployment key。

实现方式可以是 `.env`、构建脚本、原生 build flavor / scheme、远程配置。敏感信息不要直接明文放进 JS bundle。

## 六、高级架构与场景设计题

### 61. 如何从 0 设计一个 RN 混合 App 架构

**答案：**

可以从以下层次回答：

- 容器层：原生负责启动、路由容器、权限、基础 SDK。
- RN 运行时层：bundle 加载、异常兜底、OTA、版本兼容。
- 通信层：统一 Native Module 协议，避免业务直接调用零散原生 API。
- UI 层：设计系统、主题、组件库。
- 数据层：网络、缓存、登录态、状态管理。
- 稳定性层：日志、埋点、崩溃、性能监控。
- 发布层：CI/CD、灰度、回滚、sourcemap。

核心取舍：哪些能力放原生，哪些放 RN，接口如何长期稳定，如何保证出问题时能定位和回滚。

### 62. 什么时候不适合使用 React Native

**答案：**

不适合或要谨慎的场景：

- 极高性能实时渲染，如大型 3D 游戏。
- 大量复杂原生页面且跨端复用价值低。
- 对启动耗时、内存、包体积极端敏感的场景。
- 团队缺乏 JS / 原生双端维护能力。
- 业务高度依赖平台专有能力。
- 安全合规要求禁止动态下发 JS。

成熟回答不是简单否定 RN，而是结合业务收益、性能边界、团队能力和长期维护成本做判断。

### 63. RN 新架构迁移怎么做

**答案：**

迁移思路：

1. 盘点当前 RN 版本、第三方库和自研 Native Modules。
2. 升级到支持新架构的 RN 版本。
3. 检查依赖是否支持 Fabric / TurboModules。
4. 为自研模块补类型声明和 Codegen。
5. 先在实验分支和内部包开启。
6. 覆盖启动、核心页面、列表、动画、原生组件。
7. 做灰度和性能对比。
8. 保留回滚方案。

风险点包括第三方库兼容、构建脚本变化、原生模块改造、线上性能指标不升反降。迁移不能只看 Demo 跑通。

### 64. 如何设计一个跨端图片组件

**答案：**

需要考虑：

- 基础 props：uri、source、resizeMode、placeholder、fallback。
- 加载状态：loading、success、error。
- 缓存策略。
- 圆角、裁剪、背景色。
- 默认占位图。
- 错误重试。
- 长列表性能。
- 图片尺寸上报和服务端裁剪。
- iOS / Android 行为一致性。
- 无障碍描述。

高级方案会把图片 URL 生成、尺寸裁剪、CDN 参数、缓存策略和业务组件解耦。

### 65. 如何设计一个 RN 页面性能治理方案

**答案：**

方案可以分为：

- 指标定义：首屏、可交互、FPS、JS 卡顿、接口耗时、内存。
- 数据采集：Native 和 JS 双端埋点。
- 基线建立：不同机型、系统、网络下的正常范围。
- 问题发现：监控、报警、性能看板。
- 定位工具：Profiler、Instruments、Perfetto、日志链路。
- 优化手段：列表、图片、bundle、初始化、网络、缓存。
- 准入机制：新增页面性能预算和上线前检查。

高级点是把性能优化从“救火”变成“工程机制”。

### 66. 如何设计一个 RN 登录态体系

**答案：**

要考虑：

- token 存储位置，敏感 token 优先安全存储。
- access token 和 refresh token。
- 登录态初始化和启动恢复。
- 请求层自动注入 token。
- token 过期刷新和并发刷新合并。
- 退出登录清理缓存、store、页面栈。
- 多端登录、踢下线和账号切换。
- 生物识别或二次验证。
- 原生和 RN 共享登录态协议。

难点在于并发请求同时遇到 401 时，不能每个请求都刷新 token，也不能让失败请求无限重试。

### 67. 如何设计 RN 离线缓存

**答案：**

设计维度：

- 缓存什么：接口数据、图片、用户草稿、配置、静态资源。
- 缓存位置：内存、AsyncStorage、MMKV、SQLite、文件系统。
- 缓存策略：TTL、版本号、LRU、按用户隔离。
- 一致性：网络恢复后同步、冲突解决。
- 安全性：敏感数据加密。
- 可观测：缓存命中率、体积、清理日志。

简单 KV 适合轻量配置，结构化大数据更适合 SQLite，频繁读写的小数据可考虑 MMKV。

### 68. 如何设计 RN 的主题和暗黑模式

**答案：**

方案要包括：

- 颜色、字号、间距、圆角等 token。
- 跟随系统和手动切换。
- 主题上下文或全局 store。
- 组件库消费 token，而不是硬编码颜色。
- 图片和图标的暗黑适配。
- 状态栏、导航栏、原生页面同步。
- 缓存用户选择。
- 避免主题切换导致整棵树过度重渲染。

高级实现会把 token、组件变体和设计规范统一维护。

### 69. RN 项目如何做模块化

**答案：**

常见模块化方式：

- 按业务 feature 拆分目录。
- 公共基础库独立 package。
- monorepo 管理多个 RN 包和工具包。
- 原生能力通过统一 bridge package 提供。
- 业务模块之间通过路由、事件或服务接口通信。
- 使用明确的依赖边界，避免循环依赖。

模块化的目标不是目录看起来复杂，而是降低协作冲突、提升复用和减少发布风险。

### 70. 如何处理 RN 和原生页面之间的数据传递

**答案：**

常见方式：

- 路由参数传递简单数据。
- 原生模块提供查询接口。
- 统一事件总线传递跨页面事件。
- 共享登录态和配置服务。
- 对复杂数据使用 id 传递，再由目标页面查询详情。

原则：

- 路由参数保持轻量。
- 不传大对象和敏感对象。
- 定义稳定协议。
- 注意页面销毁后回调和事件监听清理。

### 71. RN 中 WebView 使用有什么坑

**答案：**

常见坑：

- JS 注入安全风险。
- RN 与 H5 通信协议不规范。
- 页面加载状态和错误兜底不足。
- 文件上传、下载、定位、相机权限差异。
- Cookie 和登录态同步。
- Android 返回键处理。
- 内存占用和页面销毁。
- 白屏和加载超时。

WebView 应封装成统一容器，处理导航拦截、JSBridge、安全域名、错误页和监控。

### 72. 如何做 RN 包体积优化

**答案：**

优化方向：

- 分析 JS bundle 依赖体积。
- 移除无用库和重复依赖。
- 按需引入图标、日期、工具库。
- 压缩图片和资源。
- 避免把大 JSON、大 base64 放入 bundle。
- Android 开启 shrinkResources、R8 等。
- iOS 清理无用资源和架构。
- Hermes 字节码和 sourcemap 分开管理。

高级回答要说明：包体积包括原生二进制、资源、JS bundle，不只是 npm 包大小。

### 73. 如何设计 RN 的日志系统

**答案：**

日志系统应包含：

- 分级：debug、info、warn、error。
- 分类：网络、路由、性能、业务、原生模块。
- 上下文：用户、版本、bundle、页面、traceId。
- 本地缓存和按需上传。
- 脱敏和隐私合规。
- 线上日志采样。
- 崩溃前关键日志保留。

开发环境可以输出详细日志，线上必须控制体积和敏感信息。

### 74. RN 中如何防止重复点击

**答案：**

常见方式：

- 按钮 loading 态禁用。
- `useRef` 保存提交中状态。
- 请求层做幂等 key。
- 服务端做幂等校验。
- 对纯跳转按钮做短时间节流。

不能只依赖前端节流。支付、下单、提交表单等关键操作必须服务端幂等。

### 75. RN 中如何处理表单复杂度

**答案：**

复杂表单要考虑：

- 字段状态管理。
- 校验规则。
- 异步校验。
- 错误展示。
- 键盘遮挡。
- 草稿保存。
- 动态字段。
- 提交防重。
- 服务端错误回填。

可以使用 React Hook Form 等库降低重渲染和样板代码。大型表单应拆分字段组件，避免每次输入导致整个页面重渲染。

### 76. 如何设计 RN 的埋点系统

**答案：**

埋点系统要包括：

- 页面曝光。
- 点击事件。
- 业务转化事件。
- 性能事件。
- 错误事件。
- 统一公共参数。
- 自动埋点与手动埋点结合。
- 事件 schema 管理。
- 采样、缓存、失败重试。

难点在于路由栈、弹窗、Tab 切换、曝光去重和跨 RN / 原生页面的一致口径。

### 77. RN 中如何处理多语言国际化

**答案：**

常见方案：

- 使用 i18n 库管理文案 key。
- 按语言加载资源。
- 支持系统语言和应用内切换。
- 格式化日期、数字、货币。
- 处理 RTL 布局。
- 图片和协议类文案本地化。
- 避免拼接句子，使用带变量模板。

高级点：国际化不仅是翻译，还涉及布局长度、换行、字体、复数规则和合规文案。

### 78. RN 页面出现偶现 bug，你怎么定位

**答案：**

定位思路：

- 明确复现条件：机型、系统、版本、网络、账号、入口。
- 查看 JS 和 Native 日志。
- 检查最近发布的 JS bundle 和原生版本。
- 加埋点确认关键状态流。
- 对异步流程加 traceId。
- 检查竞态、重复请求、页面卸载后 setState。
- 尝试降级、灰度对比、回滚验证。

高级候选人会主动关注“偶现”背后的异步竞态和生命周期问题，而不是只说“加日志”。

### 79. 如何评估一个 RN 第三方库是否可用

**答案：**

评估维度：

- 是否维护活跃。
- 是否支持当前 RN 版本和新架构。
- iOS / Android 是否都稳定。
- issue 和 release 质量。
- 是否有原生依赖和隐私风险。
- 包体积和性能影响。
- API 是否清晰。
- 是否容易 fork 和修复。
- 社区使用量和替代方案。

重要基础库不要只看 star 数，要看维护质量和项目兼容性。

### 80. 如何设计一个 RN 技术升级方案

**答案：**

升级方案包括：

1. 盘点当前版本、依赖、原生改动和历史补丁。
2. 阅读目标版本 breaking changes。
3. 建立升级分支和最小可运行 demo。
4. 逐步处理 JS、iOS、Android 编译问题。
5. 修复第三方库兼容问题。
6. 跑自动化测试和核心业务回归。
7. 对比启动、内存、FPS、崩溃率。
8. 小流量灰度。
9. 准备回滚方案和问题清单。

高级答案要体现风险管理：升级不是改到能编译，而是确保上线后稳定。

## 七、面试追问题和参考判断

### 81. 让你优化一个 RN 长列表，你会先改什么

**参考答案：**

先确认问题类型：白屏、掉帧、内存高、首屏慢还是图片闪烁。然后查看数据量、item 高度是否固定、图片尺寸、renderItem 是否稳定、item 是否重复渲染。

优先动作：

- 使用 `FlatList` / FlashList。
- 稳定 key。
- 拆分并 memo item。
- 固定高度提供 `getItemLayout`。
- 图片缩略图和缓存。
- 调整批量渲染参数。
- 分页加载。

好答案会强调测量和定位，不会一上来只背参数。

### 82. JS 线程和 UI 线程都可能卡顿，怎么区分

**参考答案：**

如果 JS 线程卡，常见现象是点击回调、状态更新、JS 动画、接口回调处理延迟。原生滚动有时仍可能保持相对流畅。

如果 UI 线程卡，常见现象是原生滚动、转场、布局、绘制也掉帧，即使 JS 没有明显长任务。

可以通过 Hermes Profile、React Profiler、Xcode Instruments、Android Profiler、Perfetto 等工具结合时间线判断。

### 83. 你封装过哪些 RN 基础设施

**参考答案：**

可以从项目经验展开：

- 网络层。
- 路由层。
- 登录态。
- 埋点日志。
- 权限服务。
- 图片组件。
- 表单组件。
- 异常监控。
- OTA 发布。
- Native Module 适配层。

回答时要说明为什么封装、API 怎么设计、解决了什么问题、遇到过什么坑。

### 84. RN 项目为什么需要原生能力兜底

**参考答案：**

因为 RN 的 JS 层可能出现 bundle 加载失败、执行异常、OTA 包损坏、首屏阻塞等问题。如果没有原生兜底，用户看到的就是白屏。

原生兜底包括：

- 启动页。
- RN 容器加载超时页。
- OTA 回滚。
- 错误页。
- 原生日志和崩溃采集。

高级点：跨端方案越动态，越要有稳定的原生兜底。

### 85. 如何判断某个逻辑应该放 JS 还是 Native

**参考答案：**

判断维度：

- 是否依赖平台能力。
- 是否需要高性能或多线程。
- 是否需要动态更新。
- 是否跨端一致。
- 是否涉及安全敏感信息。
- 是否会频繁调用。
- 团队维护成本。

一般 UI 和业务流程适合 JS，高频传感器、加密、大文件、音视频、复杂原生 SDK 更适合 Native 或 JSI。

### 86. RN 项目线上出了严重 JS bug，怎么处理

**参考答案：**

处理顺序：

1. 确认影响范围、版本、bundle、用户量。
2. 看是否能通过远程配置关闭入口。
3. 如果支持 OTA，准备修复包并灰度。
4. 上传对应 sourcemap 还原堆栈。
5. 监控崩溃率、白屏率和业务指标。
6. 必要时回滚到上一稳定 bundle。
7. 复盘增加测试、灰度和准入规则。

高级回答要提到止血优先，之后再定位根因。

### 87. 为什么 RN 项目需要 sourcemap

**参考答案：**

线上 JS bundle 通常被压缩混淆，错误堆栈里的文件名、行列号不可读。sourcemap 可以把线上错误还原到源码位置。

注意点：

- sourcemap 必须和 bundle 版本严格对应。
- OTA 每个 bundle 都要保留 sourcemap。
- sourcemap 不能随意公开。
- 监控平台需要记录 binary version 和 bundle version。

### 88. React Native 和 Flutter 怎么比较

**参考答案：**

可以从以下维度比较：

- 语言：RN 使用 JS / TS，Flutter 使用 Dart。
- 渲染：RN 使用原生视图体系或新架构渲染链路，Flutter 自绘。
- 生态：RN 更贴近 React 生态，Flutter UI 一致性强。
- 性能：两者都能做好，瓶颈类型不同。
- 原生集成：RN 对已有原生 App 混合接入通常更自然。
- 团队成本：取决于团队已有技术栈。

成熟回答不要绝对化，应结合业务场景。

### 89. RN 中 Context 滥用会有什么问题

**参考答案：**

如果一个大 Context 放了很多状态，每次 value 变化都会导致所有消费该 Context 的组件重新渲染，可能造成性能问题。

优化方式：

- 拆分 Context。
- value 使用 memo。
- 高频状态不要放大 Context。
- 使用 selector 型状态库。
- 把服务端缓存交给 React Query 等工具。

### 90. RN 中如何避免页面卸载后的异步 setState

**参考答案：**

做法：

- 请求支持取消，如 AbortController。
- effect cleanup 中标记取消。
- 页面失焦或卸载时取消订阅。
- store 层处理请求生命周期。
- 避免异步回调里直接假设组件仍存在。

示例：

```tsx
useEffect(() => {
  let alive = true;

  loadData().then((data) => {
    if (alive) {
      setData(data);
    }
  });

  return () => {
    alive = false;
  };
}, []);
```

更好的方式是让请求层支持真正取消，而不是只靠标记位。

## 八、快速背诵版

### 1. RN 性能优化一句话怎么答

先定位是 JS 线程、UI 线程、列表、图片、网络还是原生模块问题，再针对性优化；不要盲目加 memo。

### 2. RN 新架构一句话怎么答

新架构主要由 JSI、TurboModules、Fabric 和 Codegen 组成，目标是减少旧 Bridge 成本，提升类型安全、渲染能力和原生通信效率。

### 3. FlatList 优化一句话怎么答

稳定 key、memo item、控制批量渲染参数、固定高度用 getItemLayout、图片做缓存和裁剪、分页加载。

### 4. Native Module 一句话怎么答

Native Module 是 JS 调用原生能力的桥接层，要处理接口设计、线程、参数校验、错误回调、生命周期和平台差异。

### 5. OTA 一句话怎么答

OTA 适合 JS 层修复和小范围业务更新，但必须做版本绑定、灰度、校验、回滚、监控和 sourcemap 管理。

### 6. RN 白屏一句话怎么答

白屏通常来自 bundle 加载失败、JS 异常、首屏阻塞、OTA 不兼容或容器初始化失败，需要 JS Error Boundary、原生兜底、监控和回滚机制一起治理。

### 7. RN 架构能力一句话怎么答

中高级 RN 工程师不只是会写页面，还要能处理跨端边界、性能稳定性、原生集成、发布回滚和长期工程治理。
